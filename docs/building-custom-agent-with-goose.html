Building a Custom Domain Agent
with Goose

A comprehensive guide to using Goose as an agent runtime for your own domain — with custom prompts, custom MCP tool servers, and zero source-code modifications. Deploy on Kubernetes as a headless service.

Overview

Goose is an open-source AI agent framework by Block. Its core is a sophisticated agent loop that orchestrates LLM reasoning, tool execution, permission checks, context management, and session persistence. While built for software engineering, the runtime is entirely configurable at the prompt and tool level — making it a powerful base for any domain agent.

The key insight: you don't need to fork Goose. All customization points are available at runtime through configuration files, environment variables, and external MCP servers.

1. Custom System Prompts

Goose uses Jinja2 templates (via minijinja) for all prompts. The template engine checks for user-provided overrides before falling back to built-in defaults: 

// prompt_template.rs — render_template()
let user_path = user_prompts_dir().join(name);  // ~/.config/goose/prompts/system.md
let template_str = if user_path.exists() {
    std::fs::read_to_string(&user_path)           // ← YOUR custom template
} else {
    CORE_PROMPTS_DIR.get_file(name)                // ← built-in fallback
};Rust

To customize, simply place your template file at the matching path inside the config directory. When GOOSE_PATH_ROOT is set, the config directory becomes $GOOSE_PATH_ROOT/config.

2. Custom MCP Servers

MCP (Model Context Protocol) servers are how you give Goose domain-specific capabilities. Each MCP server exposes tools (functions the LLM can call) and optionally resources (data the LLM can read). Write them in any language — Python, TypeScript, Go, Rust — MCP has SDKs for all.

Transport Types

Goose supports two active transport types for connecting to MCP servers:

Recommended

Streamable HTTP

HTTP + SSE-based communication. Ideal for Kubernetes where MCP servers run as separate pods with internal Service URLs. Supports headers for auth.

Alternative

Stdio

Goose spawns the MCP server as a subprocess and communicates via stdin/stdout. Useful for local development or sidecar containers.

Note
The sse transport type is deprecated and kept only for config compatibility. Use streamable_http instead.

Tool Design Principles

When designing your MCP tools, keep these principles in mind:

  • High-level, not raw CLI — Instead of exposing shell_execute("acme pipeline create --source=X --target=Y"), expose create_pipeline(source, target_layer, transforms). The MCP server handles CLI translation internally.
  • Clear tool descriptions — The LLM reads your tool descriptions to decide when and how to use them. Be precise about parameters, return values, and side effects.
  • Annotate read vs. write — MCP tools support readOnlyHint annotations. Goose's SmartApprove mode uses these to auto-approve read operations.
  • Return structured data — Return JSON so the LLM can reason about results programmatically.
  • Scope tightly — Don't expose generic shell access. Only expose the operations your agent should perform.
  • Provide instructions — Your MCP server's instructions field gets injected into the system prompt. Use it to explain workflows and best practices.

Example: Data Pipeline MCP Server

A Python MCP server for a data platform might look like this:

from mcp.server import Server
from mcp.types import Tool, TextContent
import subprocess, json

server = Server(
    name="acme-pipeline-tools",
    instructions="""Tools for managing data pipelines on the Acme platform.

Workflow:
1. Use list_data_sources to discover available sources
2. Use preview_data to inspect source schemas and sample data
3. Use create_pipeline to build transformation pipelines
4. Use get_pipeline_status to verify pipeline health

Naming convention: {source}_{layer}_{descriptor}
Layers: raw → bronze → silver → gold"""
)

@server.tool(
    name="list_data_sources",
    description="List all available data sources in the catalog",
    annotations={"readOnlyHint": True}
)
async def list_data_sources(category: str = None) -> list[TextContent]:
    cmd = ["acme", "catalog", "list"]
    if category:
        cmd.extend(["--category", category])
    result = subprocess.run(cmd, capture_output=True, text=True)
    return [TextContent(type="text", text=result.stdout)]

@server.tool(
    name="create_pipeline",
    description="Create a data transformation pipeline",
    annotations={"readOnlyHint": False}
)
async def create_pipeline(
    source: str,
    target_layer: str,   # bronze | silver | gold
    transforms: list[str],
    schedule: str = "@daily"
) -> list[TextContent]:
    config = {
        "source": source,
        "layer": target_layer,
        "transforms": transforms,
        "schedule": schedule
    }
    result = subprocess.run(
        ["acme", "pipeline", "create", "--config", json.dumps(config)],
        capture_output=True, text=True
    )
    return [TextContent(type="text", text=result.stdout)]

# ... more tools: preview_data, get_pipeline_status, deploy_agent, etc.

if __name__ == "__main__":
    server.run(transport="streamable-http", host="0.0.0.0", port=8080)Python
Key Detail
The instructions you pass to the MCP Server constructor get injected into the agent's system prompt automatically (inside the {% for extension in extensions %} loop). This is how the LLM learns your domain workflows.

3. Extension Configuration

config.yaml

Extensions are configured in ~/.config/goose/config.yaml (or $GOOSE_PATH_ROOT/config/config.yaml). Each extension entry specifies its type, connection details, and whether it's enabled: 

# config.yaml — Extensions section
extensions:

  # Your custom MCP server over HTTP (running as a separate k8s pod)
  acme-pipeline-tools:
    enabled: true
    type: streamable_http
    name: acme-pipeline-tools
    description: Data pipeline management tools
    uri: http://mcp-pipeline-tools.default.svc.cluster.local:8080/mcp
    timeout: 300

  # Another MCP server for k8s monitoring
  acme-k8s-monitor:
    enabled: true
    type: streamable_http
    name: acme-k8s-monitor
    description: Kubernetes resource monitoring (read-only)
    uri: http://mcp-k8s-monitor.default.svc.cluster.local:8081/mcp
    timeout: 120

  # A stdio-based MCP server (sidecar or local binary)
  acme-catalog:
    enabled: true
    type: stdio
    name: acme-catalog
    description: Data catalog browser
    cmd: acme-catalog-mcp
    args: ["--format", "json"]
    envs:
      CATALOG_HOST: catalog.internal.acme.com
    env_keys: [CATALOG_API_KEY]  # Pulled from secrets store

  # Explicitly disable built-in developer extension
  developer:
    enabled: false
    type: builtin
    name: developerYAML

The uri, headers, and envs fields support ${VAR} and $VAR syntax. Values are resolved from the environment at connection time. The env_keys array pulls values from Goose's secret store (keyring or secrets.yaml).

Recipes

Recipes are YAML files that define a complete session configuration — extensions, prompts, settings, and initial instructions. They're a powerful way to package your agent's configuration: 

# acme-pipeline-builder.yaml
version: "1.0.0"
title: "Acme Pipeline Builder"
description: "Builds data pipelines on the Acme platform"

instructions: |
  You are a data pipeline specialist for the Acme platform.
  Always follow the medallion architecture pattern.
  Present plans before executing. Verify results after creation.

prompt: "What data pipeline would you like to build today?"

extensions:
  - type: streamable_http
    name: acme-pipeline-tools
    uri: http://mcp-pipeline-tools:8080/mcp
    timeout: 300
  - type: streamable_http
    name: acme-k8s-monitor
    uri: http://mcp-k8s-monitor:8081/mcp

settings:
  goose_provider: anthropic
  goose_model: claude-sonnet-4-6
  max_turns: 50

retry:
  max_retries: 2
  checks:
    - type: shell
      command: "acme pipeline validate --last-created"YAML

When a recipe specifies extensions, those override the global config.yaml extensions for that session. The instructions field gets set as a system prompt override via the agent's override_system_prompt() method.

4. Permission System

Goose Modes

The mode controls how tool calls are authorized at runtime:

ModeBehaviorBest For
auto All tools execute without confirmation Fully autonomous agent (headless service)
smart_approve Read-only tools auto-approve; write tools ask Interactive use with safety guardrails
approve Every tool call requires explicit approval High-security environments
chat No tool execution, LLM chat only Planning/discussion mode

Set the mode via environment variable or config:

# Environment variable
export GOOSE_MODE="auto"

# Or in config.yaml
goose_mode: "auto"Shell / YAML
Recommendation for headless deployment
Use auto mode for headless k8s deployment where there's no human in the loop, but scope your MCP tools tightly so the agent can only perform safe operations. If you need human approval for destructive operations, use smart_approve and annotate your MCP tools with readOnlyHint.

permission.yaml

Pre-configure per-tool permissions in ~/.config/goose/permission.yaml:

# permission.yaml
user:
  always_allow:
    - acme-pipeline-tools__list_data_sources
    - acme-pipeline-tools__preview_data
    - acme-pipeline-tools__get_pipeline_status
    - acme-k8s-monitor__get_pod_status
    - acme-k8s-monitor__get_pod_logs
  ask_before:
    - acme-pipeline-tools__create_pipeline
    - acme-pipeline-tools__deploy_agent
  never_allow:
    - acme-pipeline-tools__delete_pipeline

smart_approve:
  always_allow:
    - acme-pipeline-tools__list_data_sources
  ask_before:
    - acme-pipeline-tools__create_pipelineYAML
Tool Name Format
Tool names in permission.yaml use the pattern extension_name__tool_name (double underscore separator).

Session Persistence
Goose stores sessions in SQLite at $GOOSE_PATH_ROOT/data/sessions.db. Mount a PersistentVolume at /goose/data if you need sessions to survive pod restarts. For ephemeral/stateless deployments, you can skip the PV.

8. Hints & Context Files

Hints are auxiliary context injected into the system prompt. They provide additional guidance without modifying the core prompt template.

How Hints Load

  • Global hints: ~/.config/goose/.goosehints — always loaded
  • Project hints: .goosehints or AGENTS.md in the working directory (and parent dirs up to git root)
  • Subdirectory hints: auto-discovered when tools reference files in subdirectories

Hints support file imports via @filename syntax:

# .goosehints
This agent works with the Acme data platform.
Always follow the medallion architecture pattern.

# Import API reference
@docs/pipeline-api.md
@docs/naming-conventions.md.goosehints

For containerized deployment, place a .goosehints file in the working directory that the agent uses. Hints appear in the system prompt under an "Additional Instructions" section, separate from the main system.md template.

Hints vs. System Prompt
Use system.md for core agent identity and behavior. Use .goosehints for project/workspace-specific context that may change per deployment or working directory.

9. Agent Loop Internals

Understanding the agent loop helps you design effective MCP tools and prompts:

Agent.reply() Loop User Message | v [Auto-compact check] ----> Summarize history if > threshold | v [Build system prompt] ----> system.md template + extensions + hints | v [Stream LLM response] ----> Provider.stream(conversation, tools) | v Tool calls requested? ----no----> Return response to user | yes | v [Security Inspector] ------> Check for injection attacks | v [Permission Inspector] ----> Check permission.yaml + GooseMode | v [Repetition Inspector] ----> Detect tool call loops | v [Execute tools] -----------> Dispatch to MCP servers | v Collect results, loop back to LLM | (max_turns check)

Context Compaction

When conversation history approaches the model's context limit, Goose automatically summarizes older messages. Controlled by GOOSE_AUTO_COMPACT_THRESHOLD (default: 0.8 = 80%).

  • Messages are summarized using the LLM itself (via compaction.md template)
  • Original messages become user-visible but agent-invisible
  • Summary becomes agent-visible but user-invisible
  • Most recent user message is always preserved
  • The compaction prompt preserves: user goals, technical context, file/code references, pending tasks
Customizing compaction for your domain
Override compaction.md to ensure domain-specific context survives compaction. For example, ensure pipeline IDs, data source references, and transformation specs are always retained in the summary.

Tool Inspection Pipeline

Every tool call passes through three inspectors in sequence:

  1. Security Inspector — Detects prompt injection and malicious inputs
  2. Permission Inspector — Checks against permission.yaml and the current GooseMode
  3. Repetition Inspector — Detects if the agent is stuck in a tool-call loop

If any inspector rejects a tool call, the agent receives an error message and can adjust its approach. This is important for safety: even in auto mode, the security inspector still runs.

10. Exposing via ACP (Agent Client Protocol)

ACP lets other applications interact with Goose as a callable agent service. The goose-acp-server binary exposes an HTTP/WebSocket API: 

# Start ACP server with no built-in extensions
goose-acp-server \
  --host 0.0.0.0 \
  --port 3284Shell

ACP Endpoints

MethodPathDescription
POST/acpSend message / initialize session
GET/acpSSE stream for responses
DELETE/acpTerminate session

Sessions are tracked via the Acp-Session-Id header. On initialization, the ACP server reads your config.yaml, loads configured extensions, and applies GOOSE_MODE, GOOSE_PROVIDER, and other settings.

Goosed REST API vs. ACP
goosed (goose-server) exposes a richer REST API with session management, extension hot-loading, and agent control endpoints (/sessions, /agent/start, /agent/add-extension, etc.). Choose goosed if you need more control. Choose ACP for a simpler, standardized agent protocol.

Pre-Flight Checklist

Before deploying your custom agent, verify:

1

Prompt works standalone

Test your custom system.md by placing it in ~/.config/goose/prompts/ locally and running goose. Verify the agent behaves as expected with your domain instructions.

2

MCP servers respond

Test each MCP server independently. Verify tools return structured responses and that instructions render correctly in the system prompt.

3

Extensions connect

Verify config.yaml extension entries connect to your MCP servers. Test with streamable_http URIs pointing to locally-running servers first.

4

Permissions are correct

Verify permission.yaml allows the right tools and blocks dangerous ones. Test in smart_approve mode to see which tools would require approval.

5

Built-in extensions are disabled

Ensure the developer extension (and any others you don't need) are set to enabled: false in config.yaml. Verify no shell access leaks through.

6

Container builds cleanly

Build the Docker image and run it locally with docker run, passing provider credentials as environment variables. Verify the agent starts and responds.

7

k8s connectivity

Deploy to k8s and verify the goose pod can reach MCP server pods via cluster DNS. Test the full flow: your app → goose → MCP servers → your platform CLI.

8

Context compaction works

Run a long conversation that exceeds 80% of the context window. Verify that compaction preserves your domain-critical information (pipeline IDs, data sources, etc.).
